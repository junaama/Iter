# Iter — Decision Log

The complete decision log from the system design session. Every line is a binding decision; when one of these changes, this file and the corresponding artifact change together.

## Phase 1: Requirements
- **Product**: Iter, distributed at iter.dev. Mac app for teams using coding agents.
- **Distribution shape**: daemon + menubar dashboard + CLI binary (`iter suggest`, etc.). Not a terminal replacement.
- **Prompt-refinement trigger**: mix of (a) skill.md-instructed agent calls to `iter suggest` and (c) post-prompt-send monitoring with user-facing suggestion. Shell wrapper deferred.
- **Primary buyer**: team (eng/ops manager, 20–200 person company). Solo = free seed tier; gets self-only learning pool.
- **Capture inputs**: user prompt, agent session details, custom system prompts, tool calls, MCPs used, wall time, subagent prompts, turns, git blame, effort tier, tool modes (chrome, browser_use, computer_use).
- **Outputs to cloud**: trufflehog-stripped JSON logs; raw source code never syncs.
- **Supported harnesses at v1**: Claude Code, Codex, Pi, OpenCode, Gemini CLI. Pattern matching on session files; no formal integrations required.
- **Latency budget for `iter suggest`**: ≤1s P99. Capture write path async. Scoring async, batched nightly + on-demand refresh, cache-preferred.
- **Availability target**: 99.0%.
- **Consistency**: eventual; exact on refresh.
- **Read/write ratio**: 40:60.
- **Scale targets**: 30 (v1) / 500 (M1) / 1,500–5,000 (M3) / 10K–25K (M6) engineers.
- **Compliance posture**: local-first ingestion → trufflehog scan → three-tier classification (clean/strippable/dirty) → only clean+redacted syncs to cloud. Postgres RLS for tenant isolation. DPA + open-sourced scanner config as interim trust. SOC 2 deferred.

## Phase 2: Capacity math
- **Architecture sizing**: 5K-engineer ceiling. Documented migration path to 25K. Do not over-build for 25K on day one.
- **Turns/session avg**: 8. Storage = ~21KB/session compressed. ~3.15 GB/day, ~95 GB/month, ~1.1 TB/year.
- **Vector store**: pgvector for v1. Migration to Pinecone/Turbopuffer triggered at ~10M vectors or HNSW rebuild pain.
- **Retention**: 90 days full traces hot in Postgres; scored summaries indefinitely; archive to Cloudflare R2 thereafter (S3-compatible API; zero egress fees; free-tier guardrail in `deploy.md`).
- **Models**: cheap tier (Haiku / Gemini Flash / Qwen / Sonnet-mini-tier) for `iter suggest` and scoring. Multi-provider routing planned to capture startup credits (OpenAI, Anthropic, Google).
- **Pricing posture**: charging from day 1. Bootcamp cohort (30 engineers) = free design-partner tier. All other users pay seat pricing. Seat revenue not expected to fully cover cost in v1, but offsets the burn.
- **Daemon transport**: persistent WebSocket. Server in Go for 3K–5K concurrent connections.
- **DB connection model**: PgBouncer transaction mode, ~50 server-side connections, separate pools for capture / query / scoring.

## Phase 3: Data layer
- **Server language**: Go.
- **Storage v1**: Postgres + pgvector + Redis. No separate vector DB.
- **Tenant isolation**: Postgres RLS on every tenant-scoped table.
- **Access patterns**: capture write, suggest read, scoring batch, dashboard read, git/outcome write, stack write, stack read.
- **Subagents**: rows in `sessions` table with self-referential `parent_session_id` FK.
- **Session lifecycle**: event-sourced child table (`session_events`). Sessions row is identity; events table is mutable lifecycle and outcome data.
- **Session detail envelope (issue 037)**: `GET /v1/sessions/:id` returns outcomes as a sibling `outcomes[]` array rather than joining them onto the session object, because a session can have multiple downstream effects and the UI needs to render them independently. Events are paged by an opaque `(occurred_at,id)` cursor and capped at 10K rows per response; subagents are fetched by one recursive CTE through depth 5 with a `subagents.truncated` flag when deeper descendants exist.
- **Stacks table**: separate from sessions. Subject to trufflehog scan + 3-tier classification. `notes: text` replaces unbounded JSON.
- **Stack sharing model**: lightweight simulator — captures wrapped solutions (harnesses, skills, doc references) NOT raw configs or env values. Scope = team only. Granularity = whole team or individual teammate. On share, user can deselect any included file reference.
- **Stack raw-config heuristic (issue 038)**: stack create requests run trufflehog classification first, then reject any decoded string field matching `[A-Z][A-Z0-9_]{4,}=`. This intentionally catches env-var-shaped snippets such as `OPENAI_API_KEY=...` even when a detector misses the value, preserving the invariant that stacks store wrapped solutions rather than raw configs, env values, secrets, or MCP credentials.
- **`suggestion_hash` formula** (migration 0003 + `internal/db/repo/suggestions.go: SuggestionHash`): `sha256(tenant_id_binary || source_prompt_utf8)` where `tenant_id_binary` is the 16-byte form of the UUID (i.e. `uuid.MarshalBinary()` in Go; `decode(replace(id::text,'-',''),'hex')` in SQL). Used as the UNIQUE dedup key for `UpsertSuggestion`'s `ON CONFLICT` path so two concurrent `/v1/suggest` requests for the same `(tenant, prompt)` collapse to one row with `hit_count` incremented. The hash is tenant-scoped so two tenants who submit the identical prompt do not collide (defense in depth — RLS already isolates rows, but the unique key would otherwise be a covert cross-tenant signal).
- **`archive_pointers.tenant_id` has NO FK to `tenants(id)`** (migration 0001 lines 214-219): intentional. The archive must survive a tenant delete so compliance forensics (subpoena, GDPR audit, ex-employee dispute) can still resolve a session UUID to its R2 object URI long after the parent tenant row is gone. RLS still enforces tenant isolation at read time; only the cascade is suppressed.
- **Tenant invite settings column (issue 039)**: `migrations/0007_tenant_settings.sql` adds `tenants.tenant_settings jsonb` with `team_invites_enabled` and `invite_link_template`. This stays on `tenants` instead of a new tenant-scoped settings table because the v1 dashboard only needs one small read-only settings object, and `tenants` is already the non-RLS lookup table guarded by the request-path admin role check. If tenant-admin settings grow beyond invite visibility, split this into a dedicated table with tenant_id + RLS in the same migration that introduces write APIs.

## Phase 4: Compute layer
- **v1 backend**: one Go binary. One Postgres. One Redis. Modal for GPU/LLM-heavy nightly scoring. Railway for the app + cron + webhooks.
- **Durable queue**: Redis Streams. No Kafka, no NATS, no SQS.
- **Nightly scoring**: Modal (GPU-ready for future fine-tuned scorer). Other scheduled jobs on Railway cron.
- **Failure modes**:
  - WS gateway crash: daemon buffers to local SQLite WAL, replays on reconnect.
  - Postgres down: `iter suggest` falls back to Redis-cached suggestions. Capture buffers in Redis Streams (24h TTL).
  - LLM provider down: `iter suggest` surfaces the failure upstream; never blocks waiting for retries.
  - Post-ingestion leak detected: delete-cascade session_id everywhere; write audit-log entry.

## Phase 5: API surface
- **Wire formats**: REST/JSON for CLI + dashboard + webhooks. WebSocket for daemon ↔ cloud.
- **Naming**: "stack" (not "env"). Endpoints: `/v1/stack/me`, `/v1/stack/:user`, etc.
- **Suggest request schema**: includes `effort` (low|med|high|xhigh|max) and `tools[]` (e.g. "chrome", "browser_use", "computer_use") in session_context.
- **Confidence thresholds**: 0–0.50 suppress; 0.50–0.80 advisory; 0.80–1.00 replace.
- **`include_evidence` default**: false.
- **Rate limits**: 100/min CLI, 600/min daemon, per token.
- **Webhooks at v1**: GitHub + Linear, HMAC-verified, Idempotency-Key required.
- **Auth**: WorkOS. Device-code flow for CLI/daemon. Token stored in macOS Keychain. Token carries tenant_id claim.
- **Versioning**: `/v1/` prefix; v2 dual-runs for ≥6 months.
- **Idempotency-Key header**: required on all POST endpoints.
- **No GraphQL, gRPC, SSE streaming, public API, or NL search in v1.**

- **Mac/CLI device-code auth (issue 066)**: both the Swift app and `cmd/iter login` use WorkOS CLI Auth's device-code grant. Defaults target AuthKit User Management endpoints (`/user_management/authorize/device` and `/user_management/authenticate`), while `ITER_WORKOS_DEVICE_AUTH_URL` and `ITER_WORKOS_TOKEN_URL` can point staging at Connect/AuthKit-domain OAuth endpoints without code changes. Tokens are shared through macOS Keychain service `dev.iter.IterApp` with accounts `access_token`, `refresh_token`, and `id_token`; the Swift app and cgo-backed darwin CLI write the refresh token with `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly`. `SessionStore` decodes `sub`, `tenant_id`, `exp`, and display claims once on load/save, refreshes 60 seconds before expiry, and clears Keychain on refresh failure so the app returns to sign-in.

- **Real-time**: suggestion popover synchronous; menubar polls local daemon 5s; dashboard fetches on mount + focus + manual refresh; daemon↔cloud WebSocket.

  - "Copy to clipboard" wording, not "Replace." Suggestion never injects into terminal directly.
- **Stack profile**: pills indicating active stack, not a runtime environment switch. Simulation happens via git worktrees in chosen directories.
- **3-party handshake** for email-domain tenant suggestion: domain match suggests, admin approves, user accepts.

## Implementation decisions

- **Migration runner**: `goose` (v3). Picked over `golang-migrate` because (a) goose uses a single `.sql` file per version with `-- +goose Up`/`Down` markers, matching the `0001_initial.sql` filename called for in `ARCHITECTURE.md` §9; (b) Go-native and importable as a library so the cloud binary can embed migrations and run them on boot; (c) supports `StatementBegin/End` for DO-blocks needed for idempotent role creation. Tracked via the `goose_db_version` table.
- **Local dev DB**: `pgvector/pgvector:pg16` docker image on port 5433, owned by `make db-up`/`db-down`. Default `DATABASE_URL` points at it; override to target Railway dev.
- **`schema.sql` retired**: removed from repo root. `migrations/0001_initial.sql` is canonical. Maintaining a parallel snapshot would drift; the migration directory is the source of truth.
- **`iter_batch` role in initial migration**: kept inline in `0001_initial.sql` (wrapped in an idempotent DO block) to preserve semantic equivalence with the retired `schema.sql`. The application role (without `BYPASSRLS`) plus role-credential separation lands in a follow-up migration per issue 003.
- **Go module path**: `github.com/iter-dev/iter`. Placeholder org per issue 007; will be moved if/when the canonical GitHub org is registered. Go 1.22 declared as the minimum in `go.mod` (toolchain may be newer; module compatibility floor is 1.22).
- **Build runner**: `Makefile` (extended; no separate `Taskfile`). Already in use for migration helpers; adding `test`/`lint` targets there avoids splitting the developer entrypoint across two tools.
- **Lint config**: `.golangci.yml` enables `errcheck`, `govet`, `staticcheck` (linters) and `gofmt` (formatter) under the golangci-lint v2 schema. Minimum-viable per issue 007; not tuned.
- **Step 2 package layout**: `internal/{scoring,suggest,redact,signals,denylist}` — one package per pure-function domain that Step 2 covers. `pkg/contracts` and the rest of the Step 3 layout (`cmd/server`, `internal/db`, `internal/api`, ...) are intentionally deferred to Step 3.
- **`pkg/contracts` introduced early**: created in Step 2 (not Step 3) so that `internal/scoring` can type-check against the same wire shapes that `contracts.py` defines. Only the scoring-related types (`ScoreSignals`, `CompositeScoreInputs`, `CompositeScoreOutput`) are mirrored at this point; the rest land alongside the packages that need them. `contracts.py` remains canonical until the Go server is written.
- **Composite scoring formula (v1)**: weighted average of present signals, weights renormalized over the subset actually supplied. Weights: `durability_7d=0.25`, `durability_30d=0.15`, `peer_reuse_count=0.20`, `self_reuse_count=0.10`, `override_rate=0.10` (inverted: contributes `1 - override_rate`), `suggestion_acceptance=0.20`. Reuse counts are mapped to `[0,1)` via `1 - exp(-n/3)`. Floats are clamped to `[0,1]`; NaN is treated as a missing signal; negative reuse counts clamp to zero. `wall_time_ms`, `turn_count`, and `contributor_weight` are accepted on the input type for forward-compat but do not contribute to the composite at v1. Formula is a placeholder calibrated to satisfy the monotonicity / boundedness / determinism invariants from `ARCHITECTURE.md` §9 Step 2; the nightly Modal scorer is expected to replace it.
- **Score input validation policy**: invalid float inputs (NaN, out-of-range) are silently coerced rather than returning an error, because `Composite` is invoked from the nightly batch over historical rows where rejecting a single dirty row would block the whole tenant's scoring. NaN → missing; negative or `>1` floats → clamped to `[0,1]`; negative reuse counts → clamped to 0. Recorded here so the suggest-side decision function (issue 009) can pick a different stance (likely stricter) without ambiguity.
- **Suggestion decision function (`internal/suggest.SuggestionAction`)**: the locked confidence thresholds (`<0.50` Suppress, `0.50–0.80` Advisory, `≥0.80` Replace) live in exactly one place — `internal/suggest/suggest.go`. A grep-equivalent AST scan (`internal/suggest/literals_test.go`) walks `internal/` and fails the build if the float values `0.5` or `0.8` appear in any non-test `.go` file outside `internal/suggest/`, enforcing the "never reimplement" invariant from `CLAUDE.md`. The `Action` enum (`Suppress | Advisory | Replace`) is mirrored in `pkg/contracts/suggest.go` so the daemon, CLI, and dashboard share one set of string constants.
- **Suggest-side input policy (issue 009)**: the request path is latency-sensitive (≤1s P99) and synchronous with the user typing, so `SuggestionAction` never returns an error. Out-of-band inputs degrade to the safest action: `NaN` → `Suppress` (refuse to surface a possibly-bad suggestion), `confidence < 0` → `Suppress` (treated as 0; same reason), `confidence > 1` (including `+Inf`) → `Replace` (clamped to 1; the upstream scorer already clamps to `[0,1]`, so this is defense-in-depth for any future signal source that doesn't). This is intentionally less strict than what the scoring layer accepts: the scorer cleans up its own inputs, so by the time `SuggestionAction` sees a value it should already be in `[0,1]`; the policy above is a contract for what happens if that invariant is violated.
- **Signal aggregation `SessionEvent` shape** (issue 011): `pkg/contracts.SessionEvent` is the in-process Go projection of the wire-level `TraceEvent` from `contracts.py` plus the `session_events` table from `migrations/0001_initial.sql`. Fields: `ID string` (dedup key, opaque to the aggregator — daemon msg_id or DB row id), `SessionID string`, `ParentSessionID *string` (non-empty ⇒ subagent), `Type contracts.EventType` (mirroring the Python enum), `OccurredAt time.Time`, `Payload map[string]any`. `ParentSessionID` carries the subagent partition rather than a separate `is_subagent` bool because the wire envelope already transports the parent uuid and a redundant bool would risk drift. The closed `EventType` enum is mirrored verbatim from `contracts.py` (sixteen kinds) and matches the `session_events.event_type` CHECK constraint. `contracts.py` remains canonical until the Go server is written.
- **Signals derived in v1** (issue 011): `internal/signals.Aggregate` populates `peer_reuse_count`, `self_reuse_count`, `override_rate` (overrides / turn_completed, clamped to `[0,1]`; requires `overrides>0 AND turns>0` — zero overrides is "no signal", not a real 0.0), and `suggestion_acceptance` (accepted / (accepted + rejected); denominator>0 rule — 0 acceptances among rejections IS a real 0.0). `durability_7d` and `durability_30d` are NOT populated by per-session aggregation — they require windowed analysis across sessions and remain the nightly Modal batch's responsibility. Counters of zero surface as `nil` pointers; a present-but-zero pointer would be indistinguishable from a real observation in downstream aggregations.
- **Subagent independence in aggregation** (issue 011): `internal/signals` exposes two entry points, `Aggregate` and `AggregateSubagent`, both consuming the same `[]SessionEvent` slice. Each ignores the other partition (determined by `SessionEvent.IsSubagent()`), preserving the "subagent independent scoring" invariant from `ARCHITECTURE.md` §9 Step 5. Callers can pass an unsorted mixed slice and call both functions to obtain the two independent score-signal vectors.
- **Dedup key for replay idempotency** (issue 011): `SessionEvent.ID` is the equality key. Events with empty IDs are NOT deduplicated — the daemon WAL can produce wire messages before they receive a server-assigned id, and at that stage replays are still controlled by the WS gateway's ack protocol. Two events with the same ID accumulate once; the order in the input slice does not matter (first-seen wins, but counters are the same regardless because dedup happens before the switch).
- **Dangerous-pattern deny-list matching policy** (issue 012): the deny-list is declared as data — a slice of `{id, regex, description}` literals in `internal/denylist/patterns.go`. Regexes compile once at package init via `regexp.MustCompile`. Match scope is **shell-command boundary**, not substring: a pattern hits only when it appears at the start of input, immediately after a newline, `;`, `|`, `&`, backtick, or `$(`, optionally preceded by `sudo`/`doas`/`env`/`exec`/`nohup`/`time`. This means prose like `"the rm -rf flag is dangerous"` is **not** flagged, while `"ls; rm -rf /"` and `"echo x | rm -rf /tmp"` are. Two exceptions are intentionally **not** boundary-gated: (a) destructive SQL (`DROP TABLE`/`DROP DATABASE`/`TRUNCATE TABLE`), so injection payloads like `'; DROP TABLE users; --` still match, and (b) the fork-bomb literal `:(){:|:&};:` because the byte sequence is unambiguous in any context. Shell line-continuations (`\` + newline) inside a command are pre-normalized to a single space in `Contains`, so a `rm \\\n-rf foo` bypass collapses to the canonical form before regex evaluation. Unicode lookalikes are **not** normalized: a Cyrillic `е` in place of ASCII `e` does not match, on the rationale that the shell would not execute it as `rm` either. The returned `patternID` is opaque and intended only for security event logs; the public suggestion-output path returns only the boolean and emits a generic "suppressed" message (per `ARCHITECTURE.md` §9 Step 5 "deny-list hit → silent"). Perf budget: <5ms over 10KB input (well below the 1s suggest-path latency invariant).
- **Trufflehog version pin (issue 010)**: `3.95.3`. Recorded in `trufflehog.version` at the repo root and mirrored as `TRUFFLEHOG_VERSION` in the `Makefile`. `internal/redact` reads the pin file at test time and asserts the installed binary matches; CI is expected to install this exact version. Upgrades are deliberate: bumping the pin requires re-running the corpus tests in `internal/redact/testdata/` to catch detector regressions.
- **PII redaction policy (issue 010)**: `internal/redact.Classify` applies a small in-tree PII detector in addition to trufflehog's secret detectors. Emails (matched by `\b[a-z0-9._%+-]+@[a-z0-9.-]+\.[a-z]{2,}\b`) and US-style phone numbers are `strippable`, replaced in place with `[REDACTED_EMAIL]` / `[REDACTED_PHONE]`. Physical street addresses (heuristic: digit run + word + street-suffix) are classified `dirty` — the heuristic is too noisy to redact cleanly and full address corpora are not appropriate for cloud sync. Personal names are NOT detected: a regex over free text against code yields unacceptable false positives, and the trufflehog secret path is the higher-value control. Recorded here as the binding v1 policy; tightening (e.g. ML-based PII) is a Step 5 hardening item, not Step 2.
- **Trufflehog wrapper failure mode (issue 010)**: per `ARCHITECTURE.md` §9 Step 5 "trufflehog failure (fail-closed)", any internal failure of the wrapper — missing binary, non-zero exit, malformed JSON, timeout — returns `dirty` with the original payload bytes (never partial). The `Classification` zero value is `dirty` by design so a forgotten code path defaults to fail-closed. Errors are surfaced alongside the classification so the caller can log them; the classification itself is authoritative.
- **HTTP router (issue 028)**: `github.com/go-chi/chi/v5`. Picked over `gin` because (a) chi handlers are plain `http.HandlerFunc`, keeping middleware composable with any stdlib-shaped library; (b) the middleware sketches in issue 029 are already written against `func(http.Handler) http.Handler`, which chi accepts natively while gin would need wrapping; (c) WebSocket upgrade in issue 043 wants direct access to `http.ResponseWriter` + `*http.Request`, awkward through `gin.Context`; (d) smaller dep graph, no global state, no transitive web framework. Trade-off accepted: gin has more batteries (binding, validation, recovery built-in); we re-implement those minimally in `internal/api/middleware/` to keep lock-in low. Do not import `chi/middleware` — middleware concerns live in `internal/api/middleware/` so we can rip chi out later if we ever need to without rewriting handlers.
- **Server skeleton scope (issue 048)**: this slice ships only the §9 Step 3 package layout (`cmd/server`, `internal/{api,ws,ingest,auth,app}`) and a boot entry that binds `PORT` and drains on SIGTERM/SIGINT with a 10s budget. No chi, no pgx, no redis — those land in 028/049/050 respectively, keeping `go.mod` clean through 048. The placeholder `/` handler returns 503 with a "skeleton — handlers land in issue 028" body so any traffic reaching this binary today is loud, not silent. `/health` deliberately punted to 028 alongside the chi wiring (the issue brief's "health endpoint" line is folded into 028).
- **`Deps` location (issue 048)**: `internal/app/deps.go`, not `cmd/server/deps.go`. Rationale: api/ws handler tests will need to construct a `Deps` (with a test logger, a fake pool, etc.) and `cmd/...` packages are not importable. Putting the bag under `internal/app` lets both `cmd/server` and `internal/api` import the same type. Deps starts with `Logger *slog.Logger` and `BuildVersion string`; comment block enumerates the deferred fields (`DB`, `Redis`, `Auth`, `Modal`) so subsequent issues have a clear extension point.
- **JWT library (issue 056)**: `github.com/lestrrat-go/jwx/v2`. Picked over `github.com/golang-jwt/jwt/v5` because (a) jwx ships first-class JWKS support (`jwk.Set`, `jwk.Fetch`, `jwt.WithKeySet`) — golang-jwt requires a hand-rolled `KeyFunc` that does its own caching; (b) the verifier's `kid`-miss + stale-while-revalidate cache lifecycle composes more cleanly against `jwk.Set.LookupKeyID` than against a golang-jwt callback; (c) jwx returns typed sentinel errors (`jwt.ErrTokenExpired()`, `jwt.ErrInvalidIssuer()`) that map directly to our public `ErrExpired`/`ErrInvalidClaims`; (d) injectable `jwt.Clock` makes the verifier testable without monkey-patching `time.Now`. Trade-off accepted: jwx's dep graph is heavier (pulls `goccy/go-json`, `lestrrat-go/blackmagic`, `lestrrat-go/httprc`); the verifier is the single ingress for every authenticated request, so correctness > dep weight. `internal/auth.Verifier` does NOT use `jwk.Cache` because we want full control over the fresh / stale-while-revalidate / kid-miss-refresh state machine documented on the type. The fetcher is abstracted behind a `JWKSFetcher` function type so tests inject a fake without standing up an HTTP server.
- **Tenant + subject claim contract (issue 056)**: WorkOS-issued JWTs MUST carry `tenant_id` as a string-encoded UUID and `sub` as a string-encoded UUID. The verifier rejects tokens missing either claim (`ErrMissingTenant` / `ErrMissingSubject`) so no downstream code can ever see a `contracts.Principal` with a zero UUID. This is the enforcement point for the locked invariant in `CLAUDE.md` ("Tokens carry `tenant_id` claim"). SAML/SSO claim mapping (when an enterprise contract lands) MUST preserve these two claims — recorded here so a future WorkOS connection config can't accidentally drop them. `roles` is optional and tolerated as either `[]string` or `[]any` (JSON-decoded form).
- **Auth cache failure-mode policy (issue 056)**: `Verifier` prefers degraded auth over no auth. Network failure with a cold cache returns `ErrAuthUnavailable` so middleware can serve HTTP 503 (distinct from a real 401). Network failure with a *warm* cache returns the cached set silently — even past the fresh+stale window — because rejecting every authenticated request during a WorkOS outage would cascade into our own outage. Stale-while-revalidate window is 10m past the 1h freshTTL; beyond that the cache is "very stale" but still served if a fresh fetch fails. Acceptable risk: a key rotation issued by WorkOS during a network partition won't reach us until the partition heals. Mitigation deferred to issue 031 (middleware can layer metrics + alerting on the staleness age).
- **LLM provider chain (issue 055)**: per-tier priority is fixed at v1 — `cheap_hot`: `anthropic → google → openai → together`; `sonnet`: `anthropic → openai`; `opus`: `anthropic`. Rationale: Anthropic Haiku is the cheapest+fastest in the cheap tier given our committed credit balance and is the only provider that currently serves all three tiers; Google Flash is the cost-optimization fallback for cheap_hot; OpenAI is a quality fallback for both cheap and sonnet; Together hosts open-weights Qwen as a deep-fallback so a one-vendor cheap-tier outage cannot take suggest down. Opus is reserved to Anthropic at v1 because the chain is rarely invoked and a fallback adds dep weight without earning its keep.
- **LLM provider implementations at v1 (issue 055)**: only `anthropic` ships a real HTTP client; `openai`, `google`, `together` are stubs that return `ErrProviderNotImplemented` when an API key is set and `ErrProviderNotConfigured` when it is not. Router treats both identically and falls through. The stubs land in this slice so cmd/server's priority chain doesn't have to be rewritten when the SDKs arrive — only the `Complete` body changes. Anthropic is wired via direct HTTP to `/v1/messages` (not the official SDK) to keep the dep graph small (the SDK pulls ~30 transitive packages we don't need); the `Provider` interface is the boundary, so swapping in the SDK later doesn't touch callers.
- **LLM circuit-breaker tuning (issue 055)**: tiny in-tree breaker in `internal/llm/breaker.go` (closed → open after `FailureThreshold` consecutive failures; open → half-open after `RecoveryDelay`; half-open success → closed; half-open failure → open with fresh recovery timer). V1 defaults: `FailureThreshold=5`, `RecoveryDelay=30s`. Picked over `github.com/sony/gobreaker` to keep the dep graph minimal and because we do not need gobreaker's half-open concurrency limit or per-failure callback features at v1. State machine is the binding contract; tuning may move to per-provider in a later slice if production data shows asymmetric reliability between providers.
- **LLM router sentinel (issue 055)**: `llm.ErrAllProvidersUnavailable` is returned (wrapped via `fmt.Errorf` with the per-provider attempts log) whenever every provider in the per-tier chain is open, errored, unregistered, or tier-mismatched, or whenever `ctx` is already expired at entry. Callers in the suggest path pattern-match it with `errors.Is` and surface `no_suggestion_reason: llm_unavailable` per ARCHITECTURE.md §7. The router NEVER imposes its own retry/backoff — the caller's `ctx.Deadline` is the sole latency budget so the `iter suggest` ≤1s P99 invariant is preserved.
- **Embedding provider chain (issue 054)**: priority is fixed at v1 — `voyage → openai → google`. Voyage (`voyage-code-3`, 1536-dim) is the only real provider implementation at this slice; OpenAI and Google ship as `ErrProviderNotImplemented` stubs that match `internal/llm`'s pattern so cmd/server's chain doesn't have to be rewritten when the HTTP implementations land. Single-dimension chain (no `cheap/sonnet/opus` split) because embeddings have one quality dimension at v1 — the schema commits to `vector(1536)` and a fallback that produced a different dim would break the index. Voyage is wired via direct HTTP to `https://api.voyageai.com/v1/embeddings` (not the SDK), same dep-minimization rationale as Anthropic in issue 055.
- **Embedding domain isolation (issue 054)**: `internal/embed` does NOT share code with `internal/llm`. Breaker, router, health, and provider interfaces are duplicated rather than abstracted into a shared package. Rationale: the two routers front different vendor lists with different wire shapes, and a circuit-breaker outage on Anthropic must not influence Voyage's health (or vice versa). Sharing breaker state across unrelated dependency chains would couple their health surfaces and force a v1-premature generic-provider abstraction. Breaker tuning is the same v1 default (`FailureThreshold=5`, `RecoveryDelay=30s`) but maintained independently — issue 055's tuning is the reference; any tuning change happens in both packages or neither.
- **Embedding cache key + TTL (issue 054)**: Redis key is `embed:cache:<sha256(model || 0x00 || input)>` (lowercase hex). The `0x00` separator prevents the `"ab"+"cd" == "abc"+"d"` collision. Keyed by `model` (not provider name) so two providers exposing the same model id share a cache slot — in practice each provider has a distinct native model id, but keying on model is the more durable invariant. TTL is 7 days (`DefaultCacheTTL`); long enough to absorb repeat embeddings across a normal multi-day iteration on a task (the suggest path retries hit the same input embeddings), short enough that a Voyage model upgrade flushes naturally within a week. The `embed:cache:` prefix is bumped if the vector encoding ever changes — that's a backward-incompatible wire change.
- **Embedding cache vector encoding (issue 054)**: little-endian binary float32 with a 4-byte little-endian `uint32` length prefix. For a 1536-dim vector that's 4 + 1536×4 = 6148 bytes — about 10× more compact than JSON (`[0.123456,-0.789012,...]` runs ~60 KiB) and about 2× more compact than `encoding/gob` (gob adds type metadata each entry). The length prefix lets the decoder reject truncated payloads strictly rather than trusting the slice length alone. Encoding uses `binary.Write` from stdlib so there are zero new dependencies. JSON was rejected on wire size (10× cache memory for the same hit rate) and gob on the per-entry type-metadata overhead.
- **Embedding cache failure isolation (issue 054)**: cache reads that error are treated as MISS — never propagated. Cache writes are fire-and-forget on a goroutine with a detached context (`context.WithoutCancel`) bounded by a 2s absolute timeout so a request cancellation doesn't drop the write that the next request would benefit from. Rationale: `iter suggest`'s ≤1s P99 budget cannot tolerate a slow `SET` on the critical path; a wedged Redis must degrade to "cache disabled" rather than break embedding. The router writes to cache only on single-input requests (the dominant call shape from the embedding worker per ARCHITECTURE.md §4); multi-input batches bypass cache entirely to keep code complexity proportional to v1 caller patterns.
- **Modal SDK pin + Python version (issue 057)**: `modal==1.2.6` (latest at pin time) on Python `3.12`. The image is `modal.Image.debian_slim(python_version="3.12")` so deploys are reproducible even if Modal's default base image changes. The stub deploys at **default concurrency (on-demand, N=0)** — the `min_containers=2` warm-pool decision from Phase 8 / ARCHITECTURE.md §8 applies to the real scorer (issue 046), not the no-op stub, because keeping containers warm to return a hardcoded dict burns budget for no signal. When issue 046 lands, the decorator becomes `@app.function(min_containers=2)` and the next `modal deploy` picks it up. Picked over `QueryExecModeExec` (no caching at all) because cache_describe still caches column-type descriptions on the pgx side (client-side, per `pgx.Conn`) so result decoding stays typed and fast; picked over `QueryExecModeCacheStatement` (the pgx default) because cache_statement creates server-side named prepared statements that PgBouncer's transaction-mode pooling invalidates between client checkouts. cache_simple_protocol was rejected because it loses parameter typing. Documented in `internal/db/doc.go`; the integration test does not exercise the PgBouncer failure mode (bare Postgres tolerates either) so this stays a code-review invariant.
- **Tenant helper as the single sanctioned entrypoint (issue 049)**: `internal/db.WithTenant(ctx, pool, tenantID, fn)` is the only function in the request path that opens a Postgres transaction. - **Repository package layout (issue 051)**: one file per table under `internal/db/repo/` exposing free functions, not methods on a `*FooRepo` struct. The worker brief sketched the latter shape but every repo function already takes the active `pgx.Tx` so there's no per-repo state to hang off a receiver; a receiver would also invite an `*pgxpool.Pool` field that re-opens the RLS-bypass surface this package exists to close. Storage types live in `internal/db/repo/` (`Tenant`, `User`, `TenantUser`, `Session`, `SessionEventRow`) and stay deliberately distinct from `pkg/contracts` (the wire layer); the API layer maps between them. Functions return `pgx.ErrNoRows` (wrapped via `fmt.Errorf("repo.<table>.<op>: %w", err)`) as the canonical "missing-or-forbidden" sentinel — callers cannot distinguish "doesn't exist" from "hidden by RLS," and that's intentional.
- **Repo function signature (issue 051)**: `(ctx, tx pgx.Tx, ...) (T, error)` everywhere. `pgx.Tx` rather than `db.QuerierIface` because (a) the request path always runs inside `db.WithTenant`/`db.WithBatch`, which already hand the caller a `pgx.Tx`; (b) a `*pgxpool.Pool` satisfies `QuerierIface` and would silently bypass RLS — a `pgx.Tx`-typed parameter makes that a compile-time error. Tests open transactions manually (for non-RLS tenancy tables) or through `db.WithTenant` (for sessions / session_events), so the contract is identical to production. `db.Querier(ctx)` from issue 049 stays available for future cross-cutting helpers but is not how repository functions consume the handle.
- **Shared `internal/db/dbtest` harness (issue 051)**: factored out so issues 052 and 053 inherit the container/migration boot. Each `dbtest.Setup` call spins its own `pgvector/pg16` container — slow (~4s per test) but simpler than a process-wide singleton and side-steps the leaky-state failure mode of "one container, many tests." `make test-rls` keeps covering both `rls_test.go` and the new repo tests under the same `integration` build tag (they both live under `./internal/db/...`); a separate `test-repo` Makefile target was rejected as duplicate boilerplate — it would just be the same DOCKER_HOST/RYUK plumbing pointing at a narrower path.
- **DB driver for Modal batch (issue 046)**: `psycopg[binary]==3.2.3` (synchronous). Picked over `asyncpg` because (a) the nightly batch has no concurrency requirement — it is a single-writer cron, not a request-path service; (b) psycopg3 ships first-class `jsonb` + `Cursor.fetchone/fetchall` with native types and stdlib-shaped APIs, while asyncpg's binary protocol forces explicit codecs for `uuid`/`jsonb` and would multiply the test surface for no latency win; (c) `[binary]` ships pre-built wheels so the Modal image build is fast and does not need `libpq-dev` at build time; (d) the smaller dep graph keeps the Modal image cold-start under ARCHITECTURE.md §8's budget. Documented at `modal/requirements.txt`.
- **Scorer version naming convention (issue 046)**: every row in `session_scores` is tagged `scorer_version = v1-modal-<sha>` where `<sha>` is the first 12 chars of `MODAL_DEPLOY_SHA`, falling through to `GIT_SHA`, then `RAILWAY_GIT_COMMIT_SHA`, and finally the literal `v1-modal-dev` for unconfigured local runs. `v1-` is the schema-of-the-formula version (locked at issue 008); `modal-` marks the scorer-binary identity (vs. Go's request-path `Composite`); `<sha>` makes a redeploy a fresh version even if the formula constants did not change, so a regression can be rolled back by deploying the prior commit and re-running the batch. Truncation to 12 chars matches `git describe`'s default short-sha length. `session_scores(session_id, scorer_version)` is the UNIQUE constraint (migration 0004, issue 052) backing ON CONFLICT, so two scorer-version strings must NEVER decode to the same logical scorer.
- **Scoring batch runs table (issue 046)**: new ops table `scoring_batch_runs` introduced in `migrations/0005_scoring_batch_runs.sql`. No `tenant_id` and no RLS — by design. Owned exclusively by the `iter_batch` BYPASSRLS role; `iter_app` has zero grants on it. Rationale: a per-run row is not tenant-scoped (the nightly batch processes every tenant in one pass) and the request path has no legitimate read access. Reusing `audit_log` was rejected because (a) `audit_log.event_type` is a closed enum and adding `scoring_batch_failed` would balloon every consumer's switch statement, and (b) the rich per-run state (`sessions_scored`, `errors_count`, `last_event_at`, `details jsonb`) does not fit `audit_log`'s shape. The partial index `idx_scoring_batch_runs_finished WHERE status='succeeded'` is the hot path — every Modal invocation reads it once to derive the new-events watermark.
- **Windowed signals owner: Modal (issue 046)**: `durability_7d` and `durability_30d` are populated EXCLUSIVELY by the nightly Modal batch. `internal/signals.Aggregate` deliberately leaves both `nil` because they require cross-session lookups (the user's other sessions within the window). Go's request-path `iter suggest` therefore never sees a durability signal at v1 — `Composite` renormalizes over the present subset, so the score stays well-defined. The first definition of "durability" (per `_select_durability` in `modal/scoring.py`): of the user's sessions ended in the window, the fraction whose `pr_merged` outcome was not paired with a later `pr_reverted` / `code_reverted_within_7d`. This is a placeholder ratio — the formula constants in "Composite scoring formula (v1)" above are LOCKED but the *definition* of durability is the nightly batch's owner; tuning happens in `modal/scoring.py` without re-deploying the Go binary.
- **Golden-file generation for Go/Python scoring parity (issue 046)**: `scripts/gen-golden-signals.go` (gated by the `ignore` build tag) drives the Go reference implementations of `internal/signals.Aggregate` + `internal/scoring.Composite` over a hand-curated event-fixture set and emits `modal/testdata/golden_signals.json`. The Python tests load that JSON and assert byte-identical signal aggregates + tolerance-bounded composite scores (`abs <= 1e-12`). Go is the canonical reference because both scoring slices (issues 008, 011) landed there first; the Python port must follow. Regeneration is a Makefile target (`make gen-golden-signals`) and is required any time the Go formula constants or signal definitions change — a stale fixture fails the Python suite loudly.
- **pgvector binding (issue 052)**: adopted `github.com/pgvector/pgvector-go` v0.4.0 over a hand-rolled `[]float32`-to-`text` marshaler. The library implements `sql.Scanner` + `driver.Valuer`, which means pgx can ferry `vector(1536)` columns in/out without `pgxvec.RegisterTypes` — the pool stays free of an `AfterConnect` hook and PgBouncer-transaction-mode (DECISIONS.md Phase 2) is unaffected. A thin `vecScanner` shim in `repo/session_embeddings.go` keeps the storage struct's `Vec []float32` field free of the pgvector type so callers don't have to import the binding to read a row. Rolling our own marshaler was rejected because the binary encoding (4-byte dim header + IEEE-754 big-endian floats) is non-obvious and the upstream already ships it tested against pgvector ≥ 0.5; the cost of "one more dependency" is dwarfed by the cost of replicating + maintaining the codec.
- **`SearchKNN` operator (issue 052)**: the HNSW index in `migrations/0001_initial.sql` was built with `vector_cosine_ops`. The only matching operator is `<=>` (cosine distance) — `<->` (L2) or `<#>` (negative inner product) trigger a full scan with the wrong ranking. The repo function deliberately reports `Similarity = 1 - cosine_distance` instead of leaking the distance, so callers can apply a threshold without remembering which operator the index uses. RLS is the tenant filter — `SearchKNN` does NOT add a `WHERE tenant_id = $N` clause because the surrounding `WithTenant` tx already does that via the policy. The integration test seeds 50 vectors under each of two tenants and asserts the result set never crosses the boundary.
- **Idempotency constraints for scoring/outcomes (issue 052, migration 0004)**: `session_scores` gets a hard `UNIQUE (session_id, scorer_version)` so the nightly Modal scorer (ARCHITECTURE.md §9 Step 6) can retry without writing duplicate score rows — `InsertScore` uses `ON CONFLICT DO NOTHING` + a follow-up `SELECT` rather than `ON CONFLICT DO UPDATE SET id = id` because the latter would bump `xmin` on every retry and re-fire replication. `outcomes` gets a partial `UNIQUE (session_id, outcome_type, external_ref) WHERE external_ref IS NOT NULL` so webhook handlers (issues 041/042) can absorb at-least-once delivery transparently; rows with `external_ref IS NULL` (e.g. `tests_failed` recorded by an internal sweeper) intentionally bypass dedup because the caller may legitimately want multiple entries. `InsertOutcome` uses `ON CONFLICT (...) WHERE external_ref IS NOT NULL DO NOTHING` so a duplicate webhook returns `ErrAlreadyExists` without aborting the surrounding transaction — catching the post-hoc `23505` would poison `WithTenant`'s tx and force callers to wrap the call in a savepoint they have no reason to know about.
- **dbtest helpers extended for 052**: `dbtest.go` gained `SeedSession` (mints a clean-classification session under a tenant/user pair) and `SeedScore` (raw `session_scores` insert with a chosen `scored_at`). Both run as the superuser handle so tests don't need to set up `WithTenant` purely for fixture data. The score helper writes `signals = '{}'::jsonb` to skip the marshaling step — tests that care about the `signals` shape drive `InsertScore` instead.
- **Repo symbol disambiguation (issue 052)**: `session_scores` and `outcomes` would both want `ListForSession` as the natural verb name, but package-level symbol collisions force a rename. Picked `ListScoresForSession` / `ListOutcomesForSession` (and `LatestScoreForSession`, `CountOutcomesByTypeSince`, `GetEmbeddingForSession`) over import aliasing — the repo package is consumed by every handler, and forcing every consumer to alias on import would be worse than the slightly longer symbol names. The session repo's `ListSessions` / `GetSession` etc. already follow the type-suffixed shape, so this is consistent with the existing surface.
- **Rate-limit middleware (issue 032)**: per-token sliding-window log keyed by JWT `jti` (`ratelimit:<token_id>`), 60s window, 100/min CLI + 600/min daemon per ARCHITECTURE.md §5. **Lua script** implementation (single EVAL/EVALSHA — atomic ZREMRANGEBYSCORE + ZCARD + conditional ZADD) chosen over MULTI/EXEC because (a) EVAL serializes through Redis's script engine so two concurrent requests on the same key cannot both clear the gate, while MULTI/EXEC would need WATCH + CAS retry to get the same atomicity; (b) one RTT vs three on the happy path matters for the `iter suggest` ≤1s P99 budget; (c) go-redis falls back from EVALSHA to EVAL on NOSCRIPT, so cold replicas self-heal. The script returns `{allowed_flag, oldest_score_ms}` — a binary admit/reject keeps the middleware comparison off-by-one-free across the Lua/Go boundary. **Fail-open** policy: `rdb == nil` OR any Redis-side error (network, script load, parse) allows the request and logs `ratelimit_redis_unavailable`. Deliberate per the brief — a transient Redis outage taking the suggest path down would cascade worse than a brief over-burst. **TokenType source**: extended `contracts.Principal` with `TokenType string` populated by the WorkOS verifier from the optional `token_type` claim. Unknown / empty values default to the CLI ceiling (100/min) — never the daemon ceiling, so a misconfigured token can't burst at the higher rate. **Whitelist**: `/health` and `/v1/webhooks/*` bypass — the probe has no Principal, and webhook auth is HMAC (not JWT), so per-token enforcement has nothing to key on. **Retry-After**: integer seconds until the oldest in-window entry rolls off; ZRANGE WITHSCORES is post-processed via `string.format('%.0f', ...)` in Lua so large unix-ms timestamps round-trip as decimal strings under both live Redis and miniredis (the latter stringifies via Lua's default `tostring`, which collapses big integers to scientific notation).
- **Tenant-context middleware (issue 034)**: per-request transaction (option A from the issue brief) opened via `db.WithTenant(ctx, deps.DB, principal.TenantID.String(), fn)` — the single sanctioned entrypoint from issue 049. Picked over option B (per-statement helper with a deferred SET LOCAL on every Query/Exec) because (a) SET LOCAL is tx-scoped and option B would have to re-issue it on every call, easily forgotten; (b) v1 handlers like `iter suggest` read embeddings + scores in a single request and need both reads to see a consistent snapshot under one tx; (c) connection-pool pressure at 5K engineers is well below pgxpool's defaults — migration to per-statement is documented in `ARCHITECTURE.md` §8 as a 25K-scale move. Commit-on-2xx, rollback-on-non-2xx implementation note: net/http has no native intercept-then-forward hook, so the middleware wraps the ResponseWriter with a `statusCaptureWriter` (records the first WriteHeader / implicit 200) and returns a private `errTenantRollback` sentinel from the inner closure when the captured status is ≥400. db.WithTenant then rolls back the tx; the response body has already been flushed to the client (option (b) in the tenant.go package comment — option (a), full `httptest.ResponseRecorder` buffering, was rejected because it would put every response body on the suggest p99 path and break streaming). Accepted trade-off: a handler that 4xxs after side-effecting an external system (Stripe charge, send email) still observes those externals while the DB tx is reverted — consistent with the existing panic-during-handler behavior under `middleware.Recover`. `/health` and `/v1/webhooks/*` bypass: the probe has no Principal, and webhook handlers (issues 041/042) own their own tx scope after HMAC verification. A nil `deps.DB` falls through to the handler with a warn log so early bring-up before `DATABASE_URL` is provisioned still boots; handlers that need the DB panic loudly via `db.Querier` rather than silently bypass RLS. Whitelist + logger configured via the package-local `WithTenantSkip` / `WithTenantLogger` (names disambiguated from Auth's `WithSkip` / `WithLogger`, same package).
- **Commit-message session marker (issue 041)**: the GitHub webhook `push` handler matches commit messages against the regex `(?i)closes\s+session:\s*<uuid>` to bind a `commit_landed` outcome to a session. Picked over (a) a custom git trailer like `Iter-Session: <uuid>` — trailers are formalized in `git interpret-trailers` but most developers don't know about them and tooling support is uneven across IDEs; and (b) inferring by commit author + timestamp — too lossy (rebases, squash-merges, co-authored commits). The `Closes session:` prefix mirrors GitHub's `Closes #123` PR-keyword convention so developers can intuit the format. Case-insensitive + arbitrary whitespace handles `Closes Session:` and trailing spaces. The marker is documented for daemon-emitted commit messages (subagent commits include it automatically) and is the only matching strategy used by the `push` event handler; PR and `check_run` events match by `(repo_hash, commit_sha)` instead because their payloads carry SHAs but not message bodies.
- **`repo_hash` formula (issue 041)**: the column `sessions.repo_hash` stores `lower(hex(sha256(canonical_repo_url)))` where `canonical_repo_url` is the repo URL lower-cased and stripped of a trailing `.git`. Both `https://github.com/Acme/Widgets.git` and `https://github.com/acme/widgets` therefore hash to the same value, so a user who configured one form in the daemon and the other in their CI still matches. SHA-256 (not MD5/SHA-1) for collision resistance; we never need to reverse the hash. The webhook handler in `internal/api/handler/webhook_github.go` and the daemon (when it eventually populates `sessions.repo_hash` per ARCHITECTURE.md §9 Step 3) MUST use the same `hashRepoURL` formula — duplicated implementations on either side would silently miss matches. Mirrored by the Linear webhook (issue 042) when those events arrive without a direct session_id.
- **`pending_outcomes` table (issue 041)**: webhook events that don't match a session at receive time are buffered into `pending_outcomes` (NOT tenant-scoped — at receive time the matching tenant is unknown). `UNIQUE(source, delivery_id)` dedups redeliveries. A late-match sweeper (TODO, not built at v1) will join by `repo_hash` to `sessions` and replay into the `outcomes` table, stamping `matched_at`. 7d retention is the v1 intent but the deletion job is also a TODO — `repo.DeleteOlderThan` exists as the single call site for when that job lands. `iter_app` reads/writes, `iter_batch` reads (and may UPDATE/DELETE during the nightly sweep). RLS is NOT enabled because there is no `tenant_id` column; the table is server-side background-job state, never the request path.
- **Shared webhook receiver shape (issue 042)**: GitHub and Linear both enter through `internal/api/webhook.Handler`, which reads the raw body once with a 1 MiB cap, verifies HMAC-SHA256 in constant time, requires the source delivery header (`X-GitHub-Delivery` or `Linear-Delivery`), and uses Redis `SETNX` for a 24h per-source idempotency key (`webhook:<source>:delivery:<id>`). Redis errors are fail-open, matching the rate-limit/idempotency posture; storage dedup still protects outcomes. Source-specific handlers own event mapping only. Linear dispatches by top-level JSON `type`: `Issue` created with an incident label plus the `Closes session: <uuid>` marker writes `incident_caused` and an `incident_linked` audit entry; `Issue` updated to Done writes `incident_resolved` audit if the original incident outcome exists and leaves the historical outcome unchanged; `Comment` is debug-logged and 200-ignored until ops asks for comment-derived signals.

- **Archive cron scheduler (issue 047)**: `github.com/robfig/cron/v3` over a hand-rolled `time.Ticker` + cron-spec parser. robfig parses real crontab syntax (`0 3 * * *`), supports DST-correct UTC scheduling via `cron.WithLocation(time.UTC)`, and ships a structured `EntryID` for runtime inspection — a hand-rolled ticker would either reimplement crontab parsing (a non-trivial subset of POSIX cron's spec) or force every caller to compute "next 03:00 UTC" themselves. The scheduler is constructed with `time.UTC` location so the crontab is interpreted in UTC regardless of `TZ`. One dep, three years of maintenance, the same package the Go stdlib's `cron`-shaped third-party projects all converge on; the alternative `robfig` retired-but-popular forks (`gocron`, `cronexpr`) ship more features but at the cost of a larger surface than a one-job v1 needs. Future in-process crons either get their own `archive.Scheduler` or this grows an `Add(spec, job)` method.
- **R2 free-tier guardrail data source (issue 047)**: the archive cron polls Cloudflare's REST endpoint `GET /accounts/{id}/r2/buckets/{name}` once per run (single Class-B op against the bucket) and projects `payload_size` against `R2_FREE_STORAGE_GB`. Class A/B operations counters would require the Analytics GraphQL endpoint (`r2OperationsAdaptiveGroups`) — at v1 we leave those `Frac` fields at zero because storage saturates first (~95 GB/month projected — DECISIONS.md Phase 2 — vs. ~50K Class A ops/day estimated, well under the 1M/month free allotment). The REST endpoint shape is stable and a single roundtrip is well within the cron's runtime budget; the GraphQL endpoint is documented at a lower fidelity and would multiply the test surface for no v1 signal. The guardrail trips at any metric ≥ `R2_USAGE_ALERT_THRESHOLD` (default 80%) and the cron returns early with `r2_usage_threshold_exceeded` set on the RunStats — next-day cron picks up where it stopped because the `archived_at IS NULL` filter is the idempotency boundary, not the cron's internal cursor. The "95% hard stop / `R2_OVERAGE_OK=true` bypass" mentioned in deploy.md is intentionally deferred — v1 ships the 80% alert as the only knob; tightening to 95% with a deliberate-overage flag lands when production data shows we need a second tier.
- **S3 client library for R2 (issue 047)**: `github.com/aws/aws-sdk-go-v2/service/s3` with a custom `BaseEndpoint` + `UsePathStyle=true` against `$R2_ENDPOINT`. R2's S3 compatibility was designed against the aws-sdk-go-v2 v1.x surface; alternative libraries (`minio-go`, `goamz`) would either re-implement the AWS signature flow or require a vendored fork to override the endpoint. `UsePathStyle` is forced ON because the default virtual-hosted-style addressing fails on bucket names containing dots — picking path-style at construction time means the bucket-naming convention has one fewer constraint to remember. The cron's payload size is bounded by per-session events + embedding + scores + outcomes (well under the 5 MiB multipart threshold even for a long session), so `PutObject` is a single non-multipart request — the additional multipart upload surface is intentionally NOT wired in. The verify-r2 script (`scripts/verify-r2.sh` + `scripts/verify-r2/main.go`) uses the IDENTICAL `archive.R2Config` + `archive.NewR2Store` construction the cron uses at runtime, so a passing verify run proves the SDK + endpoint + credentials work as a quartet — one source of truth for "how Iter talks to R2."
- **Archive bundle format (issue 047)**: tar-then-zstd (one tarball containing per-shape JSON entries; the whole tarball is zstd-compressed) over zstd-of-individual-entries. zstd's dictionary builds across the whole tar payload, so repeated JSON keys like `session_id` and `tenant_id` share encoding across `session.json`, `events.json`, etc. — per-entry zstd would forgo that win on payloads dominated by structural repetition. Entries live as separate files inside the tarball (one JSON document per shape) so the archive is human-inspectable with `tar -xOf foo.tar.zst session.json | jq .` — a single combined JSON would be simpler to write but loses the ability to grep individual shapes without unmarshaling the entire blob. The tarball overhead is ~100 bytes per entry, dominated by payload size on any non-empty session. Compression level: zstd default (3); higher levels add ~30% CPU for <5% additional savings on JSON, and the cron is bottlenecked by R2 `PutObject` roundtrip, not CPU.
- **Archive cron BatchDB wiring (issue 047)**: `cmd/server` reads `DATABASE_URL_BATCH` and populates `deps.BatchDB`. A missing env var is a Warn-and-skip path (not a fatal boot), because the request path remains fully usable without the batch pool — the archive cron simply does not start, and the existing "BatchDB left nil" comment in `internal/app/deps.go` is updated accordingly. The archive scheduler also nil-checks `BatchDB` and skips its own start so the cron's failure mode in a dev/under-configured deploy is "no archive ticks" rather than "process panics at 03:00 UTC." `dbtest.NewBatchPool` is the test-side equivalent: alters the `iter_batch` role password (separate from `AppRolePassword` so an accidental swap fails loudly), opens a pool with the iter_batch DSN, and chains into the existing `TestDB.cleanupFn` so callers don't have to track a second pool lifetime. Used by the cron's integration tests under `//go:build integration`.
